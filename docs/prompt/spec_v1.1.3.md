# 護理師排班系統規格 v1.1.3

## 一、技術可行性分析

本系統屬於典型的「限制條件最佳化 + 行事曆管理」應用，在目前技術水準下完全可實作。

- 排班最佳化可透過既有演算法（如整數規劃、啟發式演算法）實現。
- 規則分層與覆寫可透過規則引擎或權重評分機制處理。
- 行事曆視圖與互動操作為成熟常見技術。
- 不需即時性極高的運算，可接受背景運算與等待結果。

**結論：技術可行，無需依賴不可取得或實驗性技術。**

## 二、初步需求整理

### 2.1 商業目標

- 降低人工排班時間與錯誤率。
- 提升排班公平性與透明度。
- 在符合法規下，盡量滿足人員偏好。

### 2.2 使用者與利害關係人

- 護理長 / 管理者（主要使用者）。
- 一般護理師（查看結果、設定偏好）。
- 醫院管理單位（制度與合規）。

### 2.3 主要使用情境

- 每月或每週進行一次排班。
- 臨時人力異動後重新計算。
- 模擬不同規則下的排班結果。

## 四、系統模組與功能規格（Spec v4）

### 4.1 行事曆模組

**目的**：讓使用者一眼掌握排班現況與未來安排。

**功能說明**：

- 月 / 週 / 日檢視切換。
- 以人員 × 時段方式呈現班表。
- 不同班別以顏色區分。
- 可點擊查看單一人員詳細班表。

**狀態與持久化**：

- 當前檢視日期與模式需被記住。
- 排班資料需可跨次開啟繼續編輯。

![行事曆檢視示意](/assets/ui/calendar-view.svg)

---

### 4.2 規則維護模組

**目的**：集中管理所有影響排班的條件。

**規則分類**：

1. **硬性規則**（不可違反）
   - 法定工時上限
   - 連續上班天數限制
   - 班別銜接限制
2. **軟性規則**（盡量滿足）
   - 夜班平均分配
   - 假日出勤平衡
3. **個人偏好**
   - 偏好班別
   - 不可上班時段

**分層式管理**：

- 全域層（法規）。
- 醫院 / 科別層。
- 個人層。

規則可被下層覆寫，但不得違反上層硬性限制。

**變動一致性**：

- 每條規則必須支援新增 / 修改 / 停用。

![規則維護示意](/assets/ui/rules-maintenance.svg)

---

### 4.3 資料維護模組

**管理內容**：

- 人員基本資料。
- 職級 / 班別 / 科別代碼。

**預設資料**：

- 系統初始化時提供合理預設班別與職級。
- 內建約 30 位測試用護理師資料（含不同年資與職級）。

**檔案體驗**：

- 匯入資料前需提供預覽。

![資料維護示意](/assets/ui/data-maintenance.svg)

---

### 4.4 排班最佳化模組

**流程**：

1. 使用者選擇排班期間。
2. 調整演算法參數（如偏好權重）。
3. 啟動最佳化計算（Streaming 顯示進度）。
4. 產生建議排班結果。

**邏輯原則**：

- 必須 100% 滿足硬性規則。
- 軟性規則以權重方式計分。
- 產出「最接近理想」的方案，而非唯一解。

**結果處理**：

- 可手動微調。
- 調整後需重新驗證規則。

![最佳化排班示意](/assets/ui/optimization.svg)

## 五、驗收條件（摘要）

- 任一硬性規則被違反時，系統必須阻止排班成立。
- 同一資料在重新開啟後狀態一致。
- 最佳化過程需可被中斷或重跑。
